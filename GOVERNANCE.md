# Project Governance

## Project Leadership

**Maintainer**: Aman Singh (@AmanSinghNp)
- Final decision authority on project direction
- Release management
- Code review and merge authority

## Decision Making Process

### Minor Changes
- Bug fixes, documentation updates, small features
- Single maintainer approval sufficient
- Community discussion encouraged but not required

### Major Changes
- New features, API changes, architectural decisions
- Require GitHub discussion with 7-day comment period
- Maintainer makes final decision after considering feedback

### Breaking Changes
- Changes that break backward compatibility
- Require RFC (Request for Comments) process
- Major version bump required

## Contributor Roles

**Contributors**: Anyone who submits a PR
**Committers**: Regular contributors with demonstrated expertise (future role)
**Maintainers**: Project leadership and release management

## Becoming a Committer

After 5+ meaningful contributions and demonstrated expertise, contributors may be invited to become committers with merge privileges.

## Code of Conduct Enforcement

Violations handled by project maintainer. Serious violations may result in temporary or permanent ban.

This governance structure may evolve as the project grows.
